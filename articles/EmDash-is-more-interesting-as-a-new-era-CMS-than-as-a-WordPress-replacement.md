tl;dr

- Cloudflare が公開した EmDash は、Astro / TypeScript / Workers / D1 / R2 で作られた CMS。
- 私は「今すぐ WordPress を置き換えるもの」ではなく、「これからの CMS の設計サンプル」として見るのがよさそうだと思っている。
- 弊社では [Find goods](https://find-goods.net/) を EmDash で、[Glow Edit](https://www.glow-edit.com/) を WordPress で運用している。
- 面白いところは、サーバ運用、プラグイン、認証の 3 つを今っぽく作り直そうとしているところ。
- 一方で、現時点では v0.1.0 preview なので、一般的な店舗サイトや企業サイトへいきなり入れるにはまだ早い。
- お客さんから「どうしても使いたい」と言われたら納品はできるが、自分から積極的には勧めない。

[EmDash](https://github.com/emdash-cms/emdash) という CMS が出ている。
Cloudflare の公式ブログでは「WordPress の精神的後継」みたいな位置づけで紹介されていて、MIT ライセンスの OSS として公開されている。

ざっくり言うと、WordPress 的な CMS を Astro、TypeScript、Cloudflare Workers、D1、R2 の時代に作り直したものだ。

弊社でも、実際に EmDash と WordPress の両方を運用している。
EmDash 側は [Find goods](https://find-goods.net/) で、WordPress 側は [Glow Edit](https://www.glow-edit.com/) だ。
なのでこの記事は、単にリリースを見て「よさそう」と言っているというより、まだ浅いながらも両方を触っている立場からの感想になる。

ただ、最初に釘を刺しておくと、私はこれを「WordPress を今すぐ倒すやつ」とは見ていない。
むしろ「AI と Cloudflare 時代に CMS を作り直すなら、こういう方向になるのかも」という技術プレビューとして見るのがちょうどよさそうだと思っている。

## 何がうれしいのか

WordPress と比べたときに、EmDash が面白いと思ったところは主に 3 つある。

1. サーバ運用の前提を変えている
2. プラグインの権限を絞ろうとしている
3. パスワードログインを前提にしていない

順番に書く。

## サーバを持ち続ける前提から離れられる

WordPress は、雑に言えば PHP と MySQL が動くサーバを用意して、そこにアプリを置いて運用する。
もちろんマネージドなホスティングを使えばかなり楽にはなる。
それでも、サーバ、PHP、DB、プラグイン、テーマ、バックアップ、WAF、アップデートの面倒を見る必要は残る。

EmDash はこの前提を少し変えている。
Cloudflare Workers で動かし、D1 を DB にして、R2 にメディアを置く構成が取れる。
アクセスがない間もサーバを立てっぱなしにするというより、必要なときに Workers で実行する発想に近い。

これは小規模サイトを大量に扱う人にはかなり魅力がある。

たとえば弊社の [Find goods](https://find-goods.net/) は、プレゼントや自分へのご褒美向けの商品情報を載せるメディアとして EmDash で動かしている。
記事一覧、カテゴリ、検索、メディア表示があり、CMS として最低限ほしいものはすでに触れる。

この手のサイトは、常に重い処理をし続けるというより、記事を公開して、読まれて、たまに更新するタイプに近い。
だから、常時サーバを持つより Cloudflare Workers に寄せる発想と相性がいい。

自分が想像している用途も、この延長線上にある。

- 中小企業向けのホームページをたくさん管理する
- 代理店単位で複数サイトを運用する
- 小さなブログや LP を大量に持つ
- 更新頻度は低いが、管理画面は必要なサイトを持つ

こういう場面では、常時サーバを管理し続けなくていい CMS という発想は強い。
Cloudflare の理解は必要なので誰にでも簡単とは言わないが、慣れている人にとってはかなりシュッとした構成にできるはず。

## プラグインに全部を渡さない設計がいい

WordPress の強みはプラグインだ。
そして、弱点もプラグインだと思っている。

WordPress のプラグインは、便利な一方でサイト全体に強い権限を持ちやすい。
1 つの脆弱なプラグインが、サイト全体やサーバ側のリスクになることがある。
長く運用している WordPress サイトほど、ここはじわじわ効いてくる。

EmDash はこの問題に正面から向き合っている。
プラグインを Cloudflare Workers のような分離環境で動かし、さらにプラグイン側が必要な権限を宣言する。
たとえば、コンテンツを読む、メールを送る、ストレージに触る、みたいな権限を最初から分ける方向だ。

Web アプリのプラグイン機構は、便利さと危険さがだいたいセットで来る。
だから「プラグインは便利だけど、何でもできてしまう」を初期設計で潰しに行くのは正しい方向だと思う。

もちろん、設計がよいことと、実際のエコシステムが安全に育つことは別の話だ。
ただ、少なくとも WordPress のつらさを知ったうえで、別の設計を試している感じがある。

## パスワード前提ではないのもよい

認証もいい。
EmDash は passkey-first、つまりパスキーを前提にしたログインを採っている。

パスワードは、結局ユーザーの行動にかなり依存する。
使い回し、弱いパスワード、パスワードマネージャーを使わない、漏洩済みパスワードを使い続ける。
運用側がどれだけ頑張っても、このあたりを完全に制御するのは難しい。

最初からパスキーを前提にするのは、かなり割り切りがある。
でも、私は CMS の管理画面ではこの割り切りはけっこう妥当だと思っている。

さらに Cloudflare Access と重ねれば、組織の認証基盤に寄せることもできる。
ここまで含めると、WordPress のログイン画面をインターネットに置いて守る、という発想とはかなり違う。

## ただし、今すぐ本番投入したいかというと別

ここまで褒めたけど、良いところだけではない。

EmDash はまだ v0.1.0 preview だ。
成熟した WordPress 代替として見るには、まだ厳しい。

WordPress には長い歴史がある。
管理画面の慣れ、プラグインの量、テーマ、ホスティング、制作者の多さ、トラブル時の検索性。
このあたりは、単純なアーキテクチャの新しさだけではひっくり返せない。

たとえば弊社の [Glow Edit](https://www.glow-edit.com/) は WordPress で運用している。
美容系の記事メディアで、カテゴリ、記事一覧、検索、投稿管理、既存テーマ、プラグインまわりの選択肢が普通に効く。
こういう「普通に記事を増やしていくメディア」では、WordPress の枯れた強さはまだかなり大きい。

一方で、[Find goods](https://find-goods.net/) を EmDash で運用していると、きれいな設計とは別に、細かい問題がいろいろ出る。
管理画面の使い勝手、権限設計、バックアップ、移行、顧客への説明、障害時の復旧手順。
CMS は「記事が保存できる」だけでは足りない。

なので、現時点で一般的な店舗オーナーやブロガーに「WordPress の代わりにこれを使いましょう」とは言いにくい。
開発者や Cloudflare に慣れた人が、未来の CMS 候補として触ってみるもの、くらいの距離感がよさそうだ。

## 私ならどう扱うか

お客さんから「どうしても EmDash で作りたい」と言われたら、たぶん納品はできる。
Astro と Cloudflare の構成は理解できるし、要件を絞れば使える場面はあると思う。

ただ、現時点で自分から積極的に勧めるかというと、まだ勧めない。
理由は単純で、CMS は作るより運用のほうが長いからだ。

小さな検証サイト、社内向け、技術に理解のある顧客、Cloudflare 前提のプロジェクト。
このあたりなら試す価値はある。

一方で、非エンジニアが日常的に更新する企業サイト、長期保守が前提の案件、プラグインで何でも解決したい案件では、まだ WordPress のほうが現実的だと思う。
Glow Edit のような記事メディアを安定して積み上げるなら、現時点では WordPress の安心感は普通に強い。

## おわり

EmDash の面白さは、WordPress の CMS / テーマ / プラグインという発想を残しながら、サーバレス、権限分離、パスキー、AI エージェント対応の時代に合わせて作り直そうとしているところにある。

「WordPress を倒すかどうか」より、「これから CMS を設計し直すなら、何を最初から変えるべきか」を見る対象として面白い。

私はそういう目線で、しばらく追ってみようと思っている。
