## はじめに

合同会社カディフ\(以下、Qadiff)では、メンバーがZennで技術記事を書くことを制限していません。

Zennは、会社公式メディアというよりも、個人が技術メモ、検証ログ、考えたこと、失敗したことを公開できる場所だと考えています。

かなりくだけて言えば、Zennは「チラシの裏」に近い場所です。

ただし、QadiffのPublicationに記事を紐づける場合、その記事は個人の記事でありながら、同時に「Qadiffに関係する人が書いた記事」として見られます。

そのため、完全な個人活動とは少しだけ扱いが変わります。

この記事では、QadiffのZenn Publicationに記事を紐づけるときの考え方と、最低限守ってほしいルールをまとめます。

## 基本方針

QadiffのZenn Publicationは、会社がメンバーに記事を書かせるための場所ではありません。

基本的には、メンバー個人が自分の関心、学び、検証、経験をもとに記事を書き、その記事を必要に応じてQadiffのPublicationに紐づける、という位置づけです。

つまり、基本方針は以下です。

```text
記事を書く主体は、書いた本人
記事の内容に責任を持つのも、書いた本人
Zenn の Publicationへの紐づけは、会社との関係性を示すためのもの
```

Qadiffとしては、メンバーが自分の名前で技術発信し、信用を積み上げていくことを大切にしたいと考えています。

## Zenn の Publicationに紐づけるメリット

Zenn の Publicationに記事を紐づけるメリットは、主に次の3つです。

### 1. Qadiffに関係する人として見てもらえる

Publicationに紐づけることで、読者からは「この人はQadiffに所属、または関係している人なのだ」と分かります。

これは、個人にとっても会社にとってもメリットがあります。

個人にとっては、会社の看板を使って信用を補強できます。

会社にとっては、メンバーがどのような技術に関心を持ち、どのような考え方で仕事をしているかを外部に伝えられます。

### 2. 個人の発信実績になる

Zennの記事は、書いた本人の実績になります。

会社のためだけの記事ではありません。

技術記事を書き、検証内容を公開し、考えを整理することは、本人のポートフォリオにもなります。

Qadiffとしても、メンバーが会社の中だけで評価されるのではなく、外部にも見える形で信用を積み上げることを歓迎します。

### 3. 収益は原則として投稿者本人に入る

Zennには、有料コンテンツやデジタルギフトなど、投稿者本人に収益が入る仕組みがあります。

Zenn の Publicationに紐づけた記事であっても、原則として、その収益は投稿者本人のものです。

これは、Qadiffの看板を使って個人が収益を得られるということでもあります。

会社としては、これを否定するつもりはありません。

むしろ、書いた人が自分の知識や経験から正当に利益を得られることは、良いことだと考えています。

## ただし、会社として守らなければいけないものがある

一方で、Zenn の Publicationに紐づける以上、会社として守らなければいけないものがあります。

たとえば、以下のようなものです。

- 顧客情報
- 案件情報
- 契約上の秘密情報
- 未公開の社内情報
- 他社のコードや資料
- 権利関係が不明な画像・図表
- 誤解を招く会社見解
- 採用・営業目的に見えすぎる表現
- 特定の個人や会社への攻撃に見える表現

Zennの記事は、気軽に書ける場所です。

ただし、インターネットに公開される以上、あとから「軽い気持ちで書きました」では済まないことがあります。

そのため、Zenn の Publicationに記事を紐づける場合は、会社として最低限の確認を行います。

これは、会社を守るためだけではありません。

書いた本人を守るためでもあります。

## Publicationに紐づける前に確認してほしいこと

Zenn の Publicationに記事を紐づける場合、公開前に以下を確認してください。

### 1. 顧客名・案件名を出していないか

顧客名、案件名、プロジェクト名、画面キャプチャ、ログ、設定値などから、特定の顧客や案件が推測できる場合があります。

明示的に名前を書いていなくても、組み合わせで分かることがあります。

不安な場合は、公開前に社内で確認してください。

### 2. 秘密情報が入っていないか

以下のような情報は、絶対に記事に含めないでください。

- APIキー
- トークン
- パスワード
- 接続文字列
- IPアドレス
- 顧客環境の構成情報
- 非公開の見積情報
- 契約条件
- 社内だけで共有されている資料

技術記事では、ログや設定ファイルを貼りたくなることがあります。

しかし、その中に秘密情報が混ざっていることがあります。

公開前に必ず社内に確認してください。

### 3. 他社の権利を侵害していないか

他社の記事、画像、図表、ソースコード、スライド、ドキュメントを使う場合は注意が必要です。

「ネットに落ちていたから使ってよい」ではありません。

引用する場合は、引用の範囲に収まっているか、出典を明記しているかを確認してください。

画像や図表は、できるだけ自分で作成してください。

### 4. 会社の公式見解に見えすぎないか

Zenn の Publicationに紐づく記事は、個人の記事であっても、会社の見解のように読まれる可能性があります。

特に、以下のような内容は注意してください。

- 特定技術への強い批判
- 特定企業への批判
- 法制度や規制に関する断定
- セキュリティ上の強い主張
- 投資、法律、税務、医療など専門責任が重い内容

個人の考えとして書く場合は、その前提が分かるようにしてください。

例：

> この記事は、筆者個人の検証メモです。会社としての公式見解ではありません。

### 5. 会社の宣伝になりすぎていないか

Zenn の Publicationは、採用広告や営業広告のためだけに使う場所ではない、という前提があります。

そのため、Publicationに紐づける記事も、技術記事として読者に価値があることを前提にします。

会社紹介や採用に関する情報を入れること自体は問題ありません。

ただし、記事の主目的はあくまで技術共有、検証共有、知見共有になるようにしてください。

## 事前レビューについて

Zenn の Publicationに記事を紐づける場合、原則として公開前に社内確認をしてください。

確認するのは、文章を細かく直すためではありません。

主な確認観点は以下です。

- 秘密情報が入っていないか
- 顧客や案件が特定されないか
- 権利関係に問題がないか
- 会社の公式見解に見えすぎないか
- 読者や第三者に不必要な攻撃性を持たないか

Qadiffとしては、メンバーの文章を過剰に管理したいわけではありません。

むしろ、自由に書いてほしいと考えています。

ただし、Publicationに紐づける以上、会社名を使うことになります。

そのため、最低限の確認は必要です。

## 個人の記事として書く場合

個人の関心で書いた記事は、原則として書いた本人のものです。

Zenn の Publicationに紐づけたからといって、会社が当然にその記事を所有するわけではありません。

記事の内容、収益、発信実績は、基本的に書いた本人に帰属します。

Qadiffは、メンバーが自分の知識や経験を自分の名前で発信することを尊重します。

## 業務として記事作成を依頼された場合

基本的に、Qadiffがメンバーに対して、業務としてZenn記事の作成を命じることは多くないと考えています。

ただし、会社の業務として記事作成を依頼する場合があります。

その場合は、通常の個人記事とは扱いが変わります。

業務として作成した記事については、事前に個別の合意を行ったうえで、原則として以下の考え方に従います。

```text
業務として作成した記事の著作権は会社に帰属する。
または、会社に無償・永続・再許諾可能な利用許諾を与える。
Zenn上の投稿・編集・削除・再掲載について会社の指示に従う。
退職後も会社は当該記事を利用できる。
```

これは、書いた人の権利を一方的に奪うためのものではありません。

会社の業務として作成した記事は、会社の資料、広報、採用、営業、ナレッジとして継続利用される可能性があります。

そのため、退職や契約終了後も、会社が必要な範囲で利用できる状態にしておく必要があります。

一方で、個人の自発的な記事まで、この扱いにするつもりはありません。

個人記事と業務記事は、できるだけ明確に分けます。

## 退職・契約終了時の扱い

退職または契約終了時には、Zenn の Publicationへの紐づけ、編集権限、管理権限を確認します。

個人が自分の意思で書いた記事については、原則として本人の記事です。

ただし、以下の場合は、会社から修正、非公開化、Publicationからの解除などを依頼することがあります。

- 秘密情報が含まれている場合
- 顧客や案件が特定される場合
- 会社の公式見解と誤解されるおそれがある場合
- 業務として作成した記事で、会社として継続管理が必要な場合
- その他、会社または第三者に不利益が生じるおそれがある場合

## 書く人にお願いしたいこと

Zenn の Publicationに記事を紐づける場合、次のことを意識してください。

- 自分の名前で書く
- 自分の経験に基づいて書く
- 分からないことを断定しない
- 秘密情報を出さない
- 他人の成果物を雑に使わない
- 会社名を使っていることを忘れない
- 読者にとって役に立つ形にする

きれいな記事である必要はありません。

完璧な記事である必要もありません。

むしろ、検証ログ、失敗ログ、判断ログのような記事には価値があります。

ただし、公開する以上、最低限の責任は発生します。

## Qadiffとして大切にしたいこと

Qadiffは、メンバーの発信を会社のものとして囲い込みたいわけではありません。

個人が技術者として信用を積み上げることは、本人にとって大切です。

そして、そういう人が所属していることは、会社にとっても価値があります。

会社の看板を使って、個人が信用を得る。

個人の発信によって、会社も信用を得る。

その関係が健全だと考えています。

ただし、会社名を使う以上、会社として守らなければいけないものもあります。

このルールは、会社が一方的に管理するためのものではありません。

書く人の自由と権利を守りつつ、会社、顧客、取引先、読者に対して無責任にならないためのものです。

## まとめ

Zenn の Publicationに記事を紐づけることは、会社の看板を少し借りながら、個人として技術発信することです。

そのため、以下を基本方針とします。

```text
個人の自発的な記事は、原則として書いた本人のもの
Publicationへの紐づけは、Qadiffとの関係性を示すもの
収益は原則として投稿者本人のもの
ただし、会社名を使う以上、秘密情報・権利・表現には注意する
業務として作成した記事は、事前合意のうえで会社利用を可能にする
```

Qadiffは、メンバーが自分の名前で技術発信することを歓迎します。

そのうえで、安心して書けるように、最低限のルールをここに定めます。
