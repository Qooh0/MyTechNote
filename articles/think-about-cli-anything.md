最近、[HKUDS/CLI-Anything](https://github.com/HKUDS/CLI-Anything) というプロジェクトが話題だ。

名前だけ見ると「なんでも CLI にするやつ」くらいの印象なのだが、少し調べてみると、これは単に便利コマンドを増やすツールというより、AI エージェントが既存ツールを扱いやすくするための CLI レイヤー、という位置づけに近い。

CLI-Anything には大きく分けて 2 つの側面がある。

ひとつは、既存アプリケーションやツール向けの CLI ラッパーを探して、導入して、起動するための `cli-hub`。もうひとつは、Claude Code や Codex などのエージェントと組み合わせて、対象ソフトウェアを解析し、エージェント向けの CLI を生成する仕組みだ。

最初に気になったのは後者だった。開発環境に最初から入れておけば、エージェントがいろいろなツールを CLI 経由で操作しやすくなるかもしれない。GUI アプリや複雑なツールも、エージェントから見れば「安定したコマンドインターフェース」になる。これはかなり魅力的だ。

ただ、devcontainer のデフォルト環境に入れるとなると、話は少し変わってくる。

自分が使っている devcontainer テンプレートは、基本的には「ソースコードを書くための環境」だ。Node.js、Python、Go、Rust などの開発に必要なものを揃え、AI コーディングエージェントが動きやすい土台を作る。ここに GIMP や Blender や LibreOffice のような実アプリ本体まで入れるつもりはない。用途が広がりすぎるし、イメージも重くなる。

CLI-Anything 自体も、実アプリ本体を置き換えるものではない。たとえば何かのアプリを操作する CLI があったとしても、そのアプリ本体が必要なら別途インストールが必要になる。つまり、devcontainer に全部入りの自動操作環境を作るというより、「必要になったときに CLI ラッパーを探せる入口」を置く、くらいがちょうどよさそうに見える。

そこで今考えているのが、`cli-anything-hub` だけを標準導入する案だ。

```bash
pipx install cli-anything-hub
```

これなら `cli-hub` コマンドが使えるようになる。たとえば、どんな CLI があるかを調べたり、必要なものだけを後から入れたりできる。

```bash
cli-hub list
cli-hub search <query>
cli-hub info <name>
cli-hub install <name>
```

この程度なら、純粋なソースコード開発環境に入れても副作用は比較的小さい。実アプリ本体を勝手に入れるわけではないし、使わなければほぼ眠っているだけの道具になる。

一方で、CLI-Anything の Codex skill や Claude plugin までデフォルトで入れるのは、まだ少し迷っている。あれは開発対象そのものに必要な依存というより、「エージェントをどう使うか」を調整する層に近い。共有の devcontainer に最初から入れるというより、各自が opt-in で足す個人設定に寄って見える。便利ではあるが、テンプレートが勝手にやることとしては少し強い。

なので、今のところの結論はこうだ。

`cli-hub` はデフォルト導入候補。  
CLI-Anything の agent skill/plugin は opt-in。  
実アプリ本体は入れない。

この線引きなら、devcontainer の軽さと汎用性を保ちつつ、「必要になったときに AI エージェント向け CLI を探せる」という余地を残せる。

AI コーディング環境は、あれもこれも入れ始めるとすぐに肥大化する。でも、入口だけを用意しておくのは悪くない。道具箱を満杯にするのではなく、道具を取りに行ける地図を一枚入れておく。CLI-Anything の `cli-hub` は、今のところそのくらいの距離感で付き合うのがよさそうだ。
